Project Outline

This page outlines a few tasks which give a guided approach to rebuilding the todo app front-end application

1. Plan your features

Looking at the supported API decide which features your todo app should include.

The example todo app includes:

  • Creating new Todo Lists
  • Deleting Todo Lists
  • Creating new Tasks
  • Deleting Tasks
  • Completing Tasks

In addition to these, there are several other features that are supported but not included in the example. You could include these in your own project as well, if you want to:

  • Renaming Todo Lists
  • Renaming Tasks
  • Moving a task from one list to another list
  • Copying a task to a new list
  • Copying an entire list

2. Create your layout

With your full feature set in mind, you can now begin to design the layout of your application.

when doing so take the following things into consideration:

  • How are Todo lists displayed to the user
  • How does the user navigate between lists
  • How does the user interact with tasks
  • How should the layout of the controls be to be intuitive

Once you've figured out the Usability Concerns, think about whether you want to support mobile as well

If you do, what changes will happen to the layout then?

After wireframing the layout of the controls

You can take a few moments to consider how you want the aestetics of your app to be.

3. Scaffold your Front-End Framework of choice

Download and setup your development environment for your front-end framework of choice.

You can utilize any framework you like. Here are a couple of suggestions that have widespread popularity.

  • Angular 2+
  • React
  • Vue
  • Ember
  • Backbone
  • Blazor (new)
  • Likely many many more I never heard of... :)

4. Implement Features

Begin implementing your features. The suggested approach is to first implement each feature independently and then bring it all together in a collective whole at the end.

For example: You might start with simply being able to get a list of all tasks from the back-end and showing them on a page.

Then you would add the ability to show only tasks for a certain todo list

Then add the ability to fetch details about the todo lists

If it seems hard to get a grasp on where to start with the features, begin with the service and implement the network calls to the api. Play with it and become familiar with its behavior

This should help build an understanding on the relationship between the API and your UI.

5. Bonus Features

It is possible to use the Front-End to implement more advanced features based on the state of your data without having to change your back-end.

Some things you could try:

  • Sort the tasks in the todo list so that uncompleted tasks are always at the top of the list
  • If you have an overview of todo lists, you could use the tasks.count to show how many tasks are in the list
  • expanding on the idea above you could also show how many are completed and uncompleted and a percentage measure
  • You could choose to hide or fade out todo lists where all tasks are completed

We're done!

If you completed all or most of the tasks outlined above you should have a working Todo App! well done!

6. Back-End Bonus Tasks!

These go well beyond the intended scope of this project

However if you are up for the challenge there are several things that can be done in the server.js file to improve on the project

The Issues

The project was left with a couple of issues in the provided API code to keep the enthusiastic learner entertained. Let's start by fixing those

  • In the routes/todo.js file fix the /api/todo/:todoId get function to utilize a join instead of doing a nested query.
  • In the routes/todo.js file fix the /api/todo/:todoId delete method to clean up the tasks table, deleting all tasks that have a reference to the now deleted todo list

The Expansion

Expand the capabilities of the API. Here are a few suggestions:

  • Add a deleted column to each table, and change the delete logic to mark rows as deleted instead of deleting them. This way data is more easily retrievable if deletions occur by mistake
  • If you do the above, change the get logic to ignore rows that are marked "deleted"
  • Add a todo_tasks with a foreign key to todos, and tasks and implement task sharing between lists in your front-end.
  • Add a taskgrouping table and add a task_group_id to the tasks table. Then implement a feature allowing for grouping tasks together (semantically sub-tasks)
  • Add authentication and authorization. This would include adding a users table, adding user_id to the todo tables and protecting the api disallowing people who are not signed in from calling it, as well as adding user sign up, login, and logout.

The above list is just suggestions. Pick and choose as you like, or do something completely different!
